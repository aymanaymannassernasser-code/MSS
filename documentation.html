⚙️ How This Works

Complete technical documentation of the Motor Starter Simulator physics engine

🎯 Core Physics Model

The simulator is built on fundamental electromechanical principles with proper understanding of motor operating regions.

1. Operating Region Limits

Synchronous Speed:
n_sync = 120 × f / poles (RPM)
Rated Slip:
s_rated = (n_sync - n_rated) / n_sync
speed_rated% = (n_rated / n_sync) × 100
Operating Region:
• Upper Limit: 99% speed (near-synchronous, minimal slip)
• Lower Limit: Speed where load torque = serviceFactor × 100%

Between these limits, motor and load torques can be equal → steady-state operation.
This is NORMAL, not a stall condition.

Service Factor: Multiplier for maximum continuous load. Default = 1.0 (100% rated load). Set to 1.15 if motor has 115% service factor rating.

Why This Matters: A motor running at 97% speed with 110% load is in its normal operating region if service factor ≥ 1.15. The simulator will NOT flag this as a stall.

2. Fundamental Equations

Newton's Second Law for Rotation:
J × dω/dt = T_net

Where:
• J = Total inertia (motor + load) in kg·m²
• ω = Angular velocity in rad/s
• T_net = Net torque = Motor torque - Load torque (Nm)
• dt = Time step = 0.01 seconds
Voltage-Current-Torque Relationships (Soft Start):
V ∝ I (linear relationship)
T ∝ V² (torque proportional to voltage squared)
∴ T ∝ I² (torque proportional to current squared)

Implementation:
voltageRatio = min(1.0, currentLimit / motorCurrentAtSpeed)
actualTorque = fullVoltageTorque × voltageRatio²
actualCurrent = fullVoltageCurrent × voltageRatio
Thermal Accumulation:
Thermal = ∫ I²dt

Capacity (hot stall):
Q_max = (LRC in amps)² × t_hotstall

Thermal percentage:
thermal% = (accumulated_I²t / Q_max) × 100

When thermal% ≥ 100% → Thermal overload stall

2. Speed Calculation

// Each iteration (dt = 0.01s):
netTorque_Nm = (netTorque% / 100) × FLT_Nm
angularAccel = netTorque_Nm / totalInertia
ω += angularAccel × dt
speed% = (ω / ω_rated) × 100

3. Curve Interpolation

The simulator uses monotonic cubic spline interpolation (Fritsch-Carlson algorithm) for smooth, physically realistic curves between the 36 data points.

// For any speed value, interpolate from table:
motorTorque% = interpolate(speed%, S_POINTS, tableMt)
motorCurrent% = interpolate(speed%, S_POINTS, tableMc)
loadTorque% = interpolate(speed%, S_POINTS, tableLt)

The 36 S_POINTS are distributed as:

  • 0-70%: Every 5% (15 points) — steady-state region
  • 70-85%: Every 3% (5 points) — transition
  • 85-100%: Every 1% (16 points) — critical near-synchronous region

🛑 Stall Detection Logic

The simulator checks 6 independent stall conditions with intelligent operating region awareness:

Operating Region Concept:
Within the operating region (lowerLimit to 99%), motor and load torques can be equal → steady-state operation.
Below the operating region: motor is accelerating → all stall checks apply.
Above the operating region: near-synchronous → only thermal applies.

1. Negative Net Torque (Always Applies)

if (motorTorque% - loadTorque% < 0) → STALL
Reason: "Insufficient Torque (Negative)"

Physical meaning: Load torque exceeds motor torque — motor physically cannot accelerate. This is checked at ALL speeds.

2. Critically Low Net Torque (Below Operating Region Only)

if (netTorque% < 0.5% AND speed < lowerOperatingLimit) → STALL
Reason: "Insufficient Torque (<0.5%)"

Physical meaning: Net torque too small to overcome friction. Only checked during acceleration phase, NOT in operating region.

3. Thermal Overload (Always Applies)

if (thermal% ≥ 100%) → STALL
Reason: "Thermal Overload"

Physical meaning: Accumulated I²t exceeds motor thermal capacity — protection trips at ANY speed.

4. Hung Motor (Below Operating Region Only)

// Checked every 1.5 seconds:
if (|Δspeed| < 0.3% over 1.5s AND time > 2s AND speed < lowerOperatingLimit) → STALL
Reason: "Hung (No Acceleration)"

Physical meaning: Motor is running but not accelerating during start-up phase.

5. Creeping (Below Operating Region Only)

if (time > 20s AND speed < lowerOperatingLimit × 0.5) → STALL
Reason: "Too Slow (Creeping - Insufficient Current)"

Physical meaning: Motor taking too long to reach halfway to operating region — insufficient starting current.

6. Timeout (Below Operating Region Only)

maxTime = user-specified (default = t_hotstall)
if (time > maxTime AND speed < lowerOperatingLimit) → STALL
Reason: "Timeout (>Xs)"

Physical meaning: Excessive starting time. User sets maximum via "Max Start Time" field (defaults to Hot Stall time).

⚠️ Key Insight: Most stall checks are disabled WITHIN the operating region because motor-load equilibrium is normal there. Only negative torque and thermal overload can occur in steady-state.

🔄 Soft Start Voltage Ramping

Current Limit Calculation

if (time ≤ rampTime):
    currentLimit% = initial% + (final% - initial%) × (time / rampTime)
else:
    currentLimit% = final%

The current limit ramps linearly from initial to final percentage over the ramp time.

Voltage & Torque Reduction

voltageRatio = min(1.0, currentLimit / motorCurrentAtSpeed)
actualMotorTorque = fullVoltageTorque × voltageRatio²
actualMotorCurrent = fullVoltageCurrent × voltageRatio

Why Torque ∝ Voltage²?

In an induction motor at constant frequency:

  • Torque = k × Flux × Current
  • Flux ∝ Voltage
  • Current ∝ Voltage (for given slip)
  • Therefore: Torque ∝ V × V = V²

This is the IEEE standard approximation for soft-start analysis.

Real-World Soft Starters vs. Our Model

Real Soft Starters (SCR/Thyristor Control):

  • Control firing angle α (when SCR triggers in each half-cycle)
  • RMS Voltage: V_rms = V_rated × sqrt((π - α + 0.5×sin(2α)) / π)
  • Voltage ramps non-linearly with α
  • Generate significant harmonics (5th, 7th, 11th, 13th...)
  • Power factor degrades during start

Our Model (Current-Limiting Approach):

  • Control current limit percentage
  • Voltage ratio = currentLimit / fullVoltageCurrent
  • Voltage ramps linearly (approximately)
  • Harmonics NOT modeled
  • Power factor NOT modeled
Why Our Approach Works:
For starting performance analysis (time, current, thermal, stalls), current-limiting gives nearly identical results to firing-angle control because:
1. Both reduce effective voltage
2. Both reduce torque by V²
3. The I²t thermal integral is the same
4. The acceleration physics (J×dω/dt = T_net) is unchanged
5. Harmonics don't affect starting torque or time

The main difference is harmonic content, which doesn't affect the key outputs users need.

Motor at 0% speed: Full-voltage current = 600%

📊 Minimum Starting Current

Algorithm

// Binary search from 200% to 700%:
for testCurrent = 200 to 700 (step 2%):
    simulate from 0% to 95% speed
    if ANY point has netTorque < 0:
        testCurrent is insufficient
    else:
        return testCurrent  // First current where all net torque ≥ 0

No safety margin is added — the returned value is the true physical minimum where net torque stays positive throughout the curve.

🔁 Time-to-Current Reverse Solver

Case 1: No Ramp (Initial == Final)

// Find single current that achieves target time:
binarySearch(200% to 700%):
    if simulationTime(current) ≈ targetTime:
        return current
// Set both initial and final to this value

Case 2: With Ramp (Initial ≠ Final)

// Preserve initial and ramp, find only final:
binarySearch(initial% to 700%):
    if simulationTime(initial, final, ramp) ≈ targetTime:
        return final
// Update only final current

How to Use

In Soft Start mode, click the START TIME result, type your target time (e.g., "5"), and press Enter. The solver automatically finds the current that achieves that time.

Limitations & Validation

Target Time Too Long:
If even at minimum current (200%) the motor starts faster than your target time, it's physically impossible. The solver alerts you.

Solution: Use a RAMP to slow down. Example: Motor starts in 1.9s at 200% but you want 5s? Set Initial 250%, Final 350%, Ramp 3s.

Target Time Too Short:
If even at maximum current (700%) the motor stalls or takes longer, the target is impossible.

Solution: Increase target time or verify motor/load curves are correct.

📈 Chart Rendering

Data Generation

// Generate 501 interpolated points (0-100% in 0.2% steps):
for speed = 0 to 100 step 0.2:
    dolMotorTorque[i] = interpolate(speed, S_POINTS, tableMt)
    dolMotorCurrent[i] = interpolate(speed, S_POINTS, tableMc)
    loadTorque[i] = interpolate(speed, S_POINTS, tableLt)
    
    if (Soft Start mode):
        currentLimit = calculate from ramp profile
        voltageRatio = min(1, currentLimit / dolMotorCurrent[i])
        ssMotorTorque[i] = dolMotorTorque[i] × voltageRatio²
        ssMotorCurrent[i] = dolMotorCurrent[i] × voltageRatio  // ACTUAL current drawn
Important: The chart plots ACTUAL current drawn, not the current limit.

Example: If limit is 450% but motor only needs 300% at 90% speed, the chart shows 300%.
As the motor accelerates, natural current decreases per the curve, so the chart will show decreasing current even with a fixed limit.

This matches real-world behavior where soft starters limit maximum current but don't force the motor to draw that much.

🎨 Curve Digitizer

Coordinate Transformation

// Canvas pixels → Data values:
fx = (canvasX - boxLeft) / boxWidth
fy = (canvasY - boxTop) / boxHeight

dataX = xMin + fx × (xMax - xMin)
dataY = useRightAxis 
    ? yRightMax + fy × (yRightMin - yRightMax)  // inverted
    : yLeftMax + fy × (yLeftMin - yLeftMax)     // inverted

Note: Y is inverted because canvas coordinates increase downward, but data values increase upward.

Control Point Storage

Control points are stored as {sp: speedPercent, val: dataValue} in data space, not canvas pixels. This makes them invariant to zoom, rotation, or canvas resize.

Curve Extraction

  1. User traces curve with finger/mouse → raw canvas pixel array
  2. Convert all pixels to data space using canvas-to-data transform
  3. For each of 36 S_POINTS speeds, linearly interpolate between traced points
  4. Store as {sp, val} array
  5. Export to simulator as integer percentages

🔢 Data Table Structure

The simulator uses 36 data points per curve (motor torque, motor current, load torque) at the following speeds:

S_POINTS = [
  0, 5, 10, 15, 20, 25, 30, 35, 40, 45, 50, 55, 60, 65, 70,
  73, 76, 79, 82, 84, 85, 86, 87, 88, 89, 90, 91, 92, 93, 94,
  95, 96, 97, 98, 99, 100
]

Each table cell contains a percentage value. The simulation reads these via:

const tableMt = [...document.querySelectorAll('.val-mt')].map(e => e.value);

⚡ Simulation Loop

Complete Algorithm

initialize: time=0, speed%=0, ω=0, thermal=0

while (time < 60s AND speed < 99%):
    // 1. Read curves at current speed
    motorTorque% = interpolate(speed%, S_POINTS, tableMt)
    motorCurrent% = interpolate(speed%, S_POINTS, tableMc)
    loadTorque% = interpolate(speed%, S_POINTS, tableLt)
    
    // 2. Apply soft start voltage reduction (if applicable)
    if (Soft Start mode):
        currentLimit% = calculate from ramp profile
        voltageRatio = min(1.0, currentLimit / motorCurrent%)
        actualMotorTorque% = motorTorque% × voltageRatio²
        actualMotorCurrent% = motorCurrent% × voltageRatio
    else:
        actualMotorTorque% = motorTorque%
        actualMotorCurrent% = motorCurrent%
    
    // 3. Calculate net torque
    netTorque% = actualMotorTorque% - loadTorque%
    
    // 4. Check ALL 6 stall conditions
    if (any stall condition met):
        STOP and report stall
    
    // 5. Update physics
    netTorque_Nm = (netTorque% / 100) × FLT_Nm
    angularAccel = netTorque_Nm / totalInertia
    ω += angularAccel × dt
    speed% = (ω / ω_rated) × 100
    
    // 6. Accumulate thermal
    current_Amps = (actualMotorCurrent% / 100) × FLC_Amps
    thermal_I²t += current_Amps² × dt
    thermal% = (thermal_I²t / Q_max) × 100
    
    // 7. Advance time
    time += dt

return: time, thermal%, minNetTorque%, finalSlip

🐛 Common Issues & Debugging

False Stalls

Symptom: Simulation reports STALL but chart shows positive net torque

Causes:

  • Table values don't match chart values — check if data was imported correctly
  • Axis assignments mixed up — verify Motor Torque and Motor Current are on correct Y-axes in digitizer
  • Load torque values too high (>150%) — indicates wrong axis scale during tracing

Axis Configuration Validation

The digitizer includes validation: if Load Torque > 150%, it warns that the curve was likely traced on the wrong axis (Current 0-700% instead of Torque 0-200%).

📝 Variable Naming Conventions

  • Pct or % suffix → Percentage value (0-700 for current, 0-300 for torque typical)
  • Nm suffix → Absolute torque in Newton-meters
  • Amps or A suffix → Current in amperes
  • RadS suffix → Angular velocity in radians/second
  • RPM suffix → Rotational speed in revolutions/minute
  • thermal → Percentage of thermal capacity used
  • thermalAbsRaw → Absolute I²t value in A²·s
